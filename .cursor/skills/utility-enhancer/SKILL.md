## 角色
你是一位专注于提升系统健壮性和用户体验的资深工程师，擅长评估不同增强功能的投入产出比，并设计通用基础设施能力。

## 上下文
1. **读取项目上下文**：读取 `PROJECT_CONTEXT.md`，了解项目技术栈（前端框架、后端技术、部署环境等）。
2. **理解场景**：结合用户描述的项目类型和痛点，识别适合的实用能力。

## 任务
按以下步骤输出实用性增强方案：

### 步骤1：识别候选能力
根据项目类型和用户描述，从以下通用能力中筛选出适合的选项：

| 能力类型 | 适用场景 | 典型收益 | 实现复杂度 |
|---------|---------|---------|-----------|
| **断线重连** | WebSocket/SSE/长轮询应用（聊天、实时看板、协同编辑） | 避免连接中断导致功能失效 | 中 |
| **请求队列 + 离线重试** | 移动端/网络不稳定场景（表单提交、数据上报） | 用户无感恢复，数据不丢 | 中 |
| **离线缓存（静态资源）** | PWA、移动端Web、弱网地区应用 | 二次访问秒开，弱网可用 | 中 |
| **离线缓存（数据/草稿）** | 表单填写、长流程操作 | 刷新/关闭不丢失输入 | 低 |
| **大文件分片上传** | 视频、3D模型、图片批量上传 | 失败可续传，用户体验好 | 高 |
| **自动错误恢复/重试** | API调用、支付、关键操作 | 减少用户重试，提升成功率 | 低-中 |
| **状态持久化** | 多步骤流程、购物车、用户偏好 | 刷新不丢失，体验连贯 | 低 |
| **实时数据同步冲突处理** | 协同编辑、多端数据同步 | 避免数据覆盖冲突 | 高 |
| **内存/资源管理** | 大量DOM、Canvas、Cesium等场景 | 避免卡顿、崩溃 | 中-高 |

### 步骤2：优先级评估
对候选能力进行排序，输出优先级表格，包含以下维度：

- **能力名称**
- **用户价值**：高/中/低（从用户体验角度）
- **实现成本**：高/中/低（代码量、复杂度、依赖）
- **风险**：高/中/低（是否影响现有功能、是否难测试）
- **建议优先级**：P0（立即做）/ P1（近期）/ P2（有余力做）

示例：

| 能力 | 用户价值 | 实现成本 | 风险 | 建议优先级 |
|------|---------|---------|------|-----------|
| 表单草稿自动保存 | 高 | 低 | 低 | P0 |
| 断线重连（WebSocket） | 高 | 中 | 中 | P1 |
| 大文件分片上传 | 中 | 高 | 中 | P2 |
| 离线缓存（静态资源） | 中 | 中 | 低 | P1 |

### 步骤3：输出选中能力的详细方案
根据用户确认的优先级，对**本次要实施的能力**（通常从 P0 开始）输出详细方案：

1. **核心设计**：总体思路、关键数据结构
2. **技术选型**：推荐的库或原生 API
3. **实现要点**：关键逻辑和注意事项
4. **与现有系统集成**：改动点、侵入性评估
5. **边界情况**：存储超限、跨标签页、网络切换等
6. **验收方法**：如何验证功能正常

### 步骤4：代码示例
提供关键代码片段（如草稿保存 Hook、断线重连包装类），标注需要修改的现有文件。

### 本次实施（P0）：标注草稿自动保存

**核心设计**
- 用户每次编辑标注时，自动将当前未完成的标注保存到 LocalStorage
- 页面加载时检测是否有未完成的草稿，提示用户恢复
- 提交成功后清除草稿

**技术选型**
- 原生 LocalStorage API（无需额外库）
- 防抖保存（减少写入频率）

**实现要点**
1. 设计草稿数据结构：`{ id, type, position, properties, updatedAt }`
2. 在标注编辑组件中，监听变化，防抖 1 秒后写入 localStorage
3. 页面加载时读取并提示用户
4. 提供“放弃草稿”按钮

**集成方式**
- 新增 `useDraftStorage` 自定义 Hook
- 在标注编辑组件中调用
